# Implementation Notes

This document explains how the Cape Media Duty Rota Management System works internally. Use these notes to understand the code and prepare for your project presentation.

---

## How the Application is Organized

### The Big Picture

```
User Opens App
      ↓
React Router checks the URL
      ↓
Route Guard checks if user is logged in
      ↓
Page Component loads
      ↓
Page uses Hooks to get data
      ↓
Hooks fetch data from Firestore (or mock data)
      ↓
Data is displayed to user
```

---

## Key Concepts to Understand

### 1. Components
Components are reusable pieces of the user interface. For example:
- `DashboardLayout` - The sidebar and top navigation used on all admin pages
- `AdminGuard` - Protects admin pages from unauthorized access
- `StaffGuard` - Protects staff pages from unauthorized access

### 2. Pages
Pages are the main screens users see. Each page is a React component:
- `AdminDashboard.tsx` - The admin home page
- `ManageStaff.tsx` - Shows the employee table
- `CreateEmployee.tsx` - Form to add/edit employees
- `CreateRota.tsx` - Form to create/edit schedules
- `StaffDashboard.tsx` - Staff member home page
- `ViewSchedule.tsx` - Staff member's personal schedule

### 3. Hooks
Hooks are special functions that manage data and state:
- `useAuth()` - Handles login, logout, and user information
- `useEmployees()` - Manages employee data (add, edit, delete, list)
- `useRotas()` - Manages rota/schedule data
- `useReports()` - Handles report generation and downloads

### 4. Routing
React Router handles navigation between pages:
```javascript
// Example routes in App.tsx
<Route path="/" element={<AdminLogin />} />
<Route path="/admin/dashboard" element={<AdminDashboard />} />
<Route path="/staff/login" element={<StaffLogin />} />
```

## How Data Flows Through the System

### Example: Adding a New Employee

1. **User fills the form** in `CreateEmployee.tsx`
2. **Form validates** the input (checks if fields are filled)
3. **Page calls** `addEmployee()` from `useEmployees` hook
4. **Hook sends data** to Firebase Firestore
5. **Firestore saves** the new employee document
6. **Hook updates** the local state with the new employee
7. **User sees** success message and is redirected

### Example: Viewing Staff Schedule

1. **Staff member logs in** at `/staff/login`
2. **System checks credentials** using `useAuth` hook
3. **Staff is redirected** to `/staff/dashboard`
4. **Dashboard calls** `getStaffRotas()` from `useRotas` hook
5. **Hook filters rotas** by the logged-in staff member's name
6. **Schedule is displayed** with upcoming and past shifts

## Firebase Firestore Structure

### Collections (Tables)

Think of collections like database tables. We have three main collections:

#### 1. `employees` Collection
Each document represents one employee:
```javascript
{
  id: "abc123",           // Auto-generated by Firestore
  name: "John Kamau",
  department: "News Room",
  email: "john@capemedia.co.ke",
  phoneNumber: "+254 712 345 678",
  createdAt: "2024-03-18T10:00:00.000Z"
}
```

#### 2. `rotas` Collection
Each document represents one duty shift:
```javascript
{
  id: "xyz789",
  staffId: "abc123",            // Links to employee
  staffName: "John Kamau",      // For easy display
  shiftDate: "2024-03-20",
  shiftTime: "morning",         // morning, afternoon, or night
  department: "News Room",
  createdAt: "2024-03-18T10:00:00.000Z"
}
```

#### 3. `reports` Collection
Each document represents one generated report:
```javascript
{
  id: "rep456",
  reportName: "Weekly Report - March 2024",
  reportType: "weekly",         // weekly, monthly, or department
  reportDate: "2024-03-18",
  generatedAt: "2024-03-18T14:30:00.000Z",
  fileType: "csv"
}
```

## How Authentication Works

### Login Flow
1. User enters email and password
2. System checks credentials against stored users
3. If valid, user info is stored in app state
4. Route guards allow access to protected pages

### Two Modes Available

**Mock Mode (Default for Demo)**
- Users are checked against a hardcoded list
- No real Firebase authentication
- Good for testing and demonstrations

**Firebase Mode (Production)**
- Uses real Firebase Authentication
- Users must be created in Firebase Console
- More secure for real-world use

### Switching Modes
In `src/lib/config.ts`:
```javascript
// For demo/presentation (uses mock data)
export const USE_MOCK_DATA = true;
export const USE_MOCK_AUTH = true;

// For real Firebase (production)
export const USE_MOCK_DATA = false;
export const USE_MOCK_AUTH = false;
```

## How Admin and Staff Flows Differ

### Admin Flow
```
Login at / → AdminGuard checks role → Dashboard
                                        ↓
                            Can access all admin pages
                            Can manage employees
                            Can create/edit rotas
                            Can generate reports
```

### Staff Flow
```
Login at /staff/login → StaffGuard checks role → Staff Dashboard
                                                      ↓
                                          Can only see own schedule
                                          Cannot manage other employees
                                          Cannot create rotas
```

### Route Guards
Route guards are components that protect pages:
```javascript
// AdminGuard.tsx
if (!user) redirect to login
if (user.role !== 'admin') redirect to staff dashboard
else show the protected page
```

## Common Questions for Presentation

### Q: How does the system prevent unauthorized access?
**A:** We use Route Guards. Before showing any protected page, the system checks if the user is logged in and has the correct role (admin or staff). If not, they are redirected to the login page.

### Q: Where is the data stored?
**A:** Data is stored in Firebase Firestore, a cloud database by Google. We have three collections: employees, rotas, and reports. Each collection stores documents (records) with specific fields.

### Q: How do staff members see only their schedules?
**A:** When a staff member logs in, we get their name from the authentication system. Then we filter all rotas to show only those where the `staffName` matches the logged-in user.

### Q: Why did you use React?
**A:** React is a popular JavaScript library for building user interfaces. It allows us to create reusable components, manage state efficiently, and build single-page applications with fast navigation.

### Q: Why TypeScript instead of JavaScript?
**A:** TypeScript adds type checking to JavaScript. This helps catch errors during development, makes the code more self-documenting, and provides better code editor support.

### Q: How does report generation work?
**A:** When the admin clicks "Generate Report," we fetch the relevant data (rotas and employees), format it as CSV (comma-separated values), and trigger a browser download. The report metadata is also saved to Firestore.

## Troubleshooting

### "Cannot read property of undefined"
- Usually means data hasn't loaded yet
- Check if loading states are being handled
- Verify Firebase connection is working

### Login not working
- Check credentials are correct
- In mock mode, verify user exists in `src/lib/auth.ts`
- In Firebase mode, check Firebase Console for users

### Data not showing
- Check browser console for errors
- Verify `USE_MOCK_DATA` setting in config.ts
- Check Firebase Firestore rules allow read access
